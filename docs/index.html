Task 1: 字符级检测实验

目标

测试多种字符级检测方案在古籍碑文场景下的效果，选出最优方案。

三个候选方案

方案 A: RapidOCR v1.4.0（主推）

  • 原理: 基于 PaddleOCR 推理引擎，v1.4.0 新增单字识别模式，返回单字坐标
  • 优点: PaddleOCR 生态成熟，无需 PyTorch，跨平台，性能好
  • 安装: pip install rapidocr-onnxruntime (ONNX 版)
  • 关键特性: 支持单字级别检测 + 识别，一次调用同时返回坐标和文字
  • 风险: 古籍碑文是否适配需要实际测试

方案 B: char-detection (fanqie03)

  • 原理: 基于 CRNN + CTC loss 的位置信息定位字符
  • 底层: chineseocr_lite OCR 引擎 + DBNet 文字检测
  • 用法: python model.py --level char --images images/0.png
  • 特点: 支持横排和竖排/古文，char 级别可逐字切分
  • 风险: 基于 chineseocr_lite，古籍场景效果未知

方案 C: CRAFT（备选）

  • 原理: Character Region Awareness for Text Detection，产出 per-character 的 region/affinity score map
  • 安装: pip install craft-text-detector（PyTorch）
  • 方案 C1: CRAFT 直接对整页检测
  • 方案 C2: PaddleOCR 行级 → CRAFT 行内
  • 风险: PyTorch 依赖重，古籍碑文背景复杂可能误检多

Baseline: 投影法（现有方案）

  • split_chars_final.py 的水平投影 + 间隙检测 + 等分补救

对比实验设计

方案模式说明
ARapidOCR 单字模式一次调用，检测+识别
Bchar-detection --level charCRNN 位置映射
C1CRAFT 整页端到端字符检测
C2PaddleOCR 行 → CRAFT 行内两级级联
Baseline投影法水平投影 + 间隙 + 等分

评估指标

指标说明
字符数检测到的区域数 vs 预期字符数
bbox 面积 CV变异系数，越小越一致
宽高比分布竖排文字应接近 1:1
字符间距标准差同列内相邻字符间距应均匀
推理时间总耗时
人工目视检查热力图/bbox 叠加图人工评估

执行顺序

  1. 方案 A: 安装 RapidOCR，查文档确认单字模式 API，在 pic1 上测试
  2. 方案 B: 安装 char-detection，在 pic1 上跑 --level char
  3. 方案 C: 安装 CRAFT，在 pic1 上跑参数扫描
  4. Baseline: 跑投影法
  5. 全量测试: 四个方案在 pic1-pic4 上跑
  6. 对比: 生成对比表，人工目视检查
  7. 结论: 选出最优方案（或组合方案）

Task 2: 色彩分析字符切分

目标

通过颜色聚类分析，在竖排文字条中找到纯色行（背景色）作为字符间的自然切分边界。

背景

现有方法（split_chars_v3/v4.py）使用水平投影法切分字符，在字符间距不明显时效果差。色彩分析法利用碑文中"字体色"和"背景色"的天然对比，找到纯背景色行作为切分间隙。

算法流程

输入: 裁剪后的文字列图片 (BGR)
  ├── 1. 颜色聚类 (Otsu 二值化)
  │     灰度化 → Otsu 自动阈值 → 前景/背景二值图
  ├── 2. 逐行颜色占比分析
  │     主色占比 = max(white_ratio, 1 - white_ratio)
  │     主色占比 >= 90% → 标记为"纯色行"
  ├── 3. 纯色行区间 → 间隔区域
  │     连续纯色行合并为间隔，间隔中点 → 切分线
  ├── 4. 降级策略
  │     找不到足够间隔 → 退化为等分方法
  └── 5. 输出: 字符裁剪图 + 可视化图

    """找纯色行区间，返回 [(start, end), ...]"""

    """完整切分流程，返回字符区域 [(y1, y2), ...]"""

关键参数

  • 聚类算法: Otsu（比 KMeans 快几个数量级）
  • purity 阈值: 0.90（90% 同色才算纯色行）
  • 当前: 仅竖排文字；架构预留横排支持

opencv-python, numpy（已在 pyproject.toml 中，无需额外依赖）

Task 3: VLM 字符完整性验证

目标

用 Qwen2.5-VL-3B via DashScope API 验证每个字符图片是否包含完整、单一的汉字。

验证范围

  • 单字: 图片中只有一个汉字
  • 完整: 字没有被截断
  • 不含其他字: 不包含相邻字符的边缘
  • 不负责识别具体是什么字（那是 HUNet 的工作）

Pipeline 位置

PaddleOCR 行检测 → 字符切分 → char crops → [可疑字符筛选] → VLM 验证 → 有效 crops → HUNet 识别

可疑字符判定条件（全部实现）

  • 面积异常: 面积 < 0.5× 或 > 1.5× 中位面积
  • 宽高比异常: 竖排字符宽高比偏离 1:1 太多
  • 切分来源: 来自等分补救的字符比来自纯色行间隙的更可疑

Prompt 设计

这个图片中是否包含一个完整的汉字？只回答 yes 或 no。
Temperature: 0.1 | Max tokens: 10

输出

只输出 pass/fail。失败的字符直接跳过，不传入 HUNet。

核心函数

def validate_character_crop(image_path) -> bool:
    """验证单个字符图片"""

def validate_character_crops(image_dir, output_dir=None, max_workers=4) -> list[dict]:

技术细节

  • 并发: ThreadPoolExecutor, 4 workers
  • API 调用: 本地调用 DashScope API（不在服务器上）
  • 重试: 指数退避 3 次 (1s, 2s, 4s)
  • 保守策略: 模糊回复默认拒绝（False）
  • 本地降级: 后期在 A100 上跑本地 Qwen2.5-VL-3B

Task 4: React 前端 MVP

目标

构建古籍 OCR 标注平台的 Web 前端，支持自动化标注监控、手动标注审校、多格式导出。

技术栈

框架React 18 + Vite + TypeScript
样式Tailwind CSS v4 (@tailwindcss/vite)
状态管理Zustand
路由react-router-dom (HashRouter)
MockMSW (Mock Service Worker)
HTTPaxios
字体加载真实字体文件
Mock 数据编造的占位数据

主题

深色主题: 基底 #1a1a1a / 面板 #242424 / 边框 #333

页面结构

1. UploadPage（上传）

  • 图片拖拽区域 + 文件选择按钮
  • 拖拽/选择后自动跳转到 DetectPage

2. DetectPage（自动化标注监控）

  • 顶部：原图预览
  • 主体：Pipeline 流程步骤列表（检测→切分→验证→识别）
  • 每步显示状态（序号/完成/进行中/待执行）+ spinner
  • 每步有参数配置入口（MVP 先做 UI 骨架）

3. AnnotatePage（手动标注与审校）— 核心

  • 左面板: 原图 + SVG 叠加层（viewBox 匹配图片自然尺寸，bbox 可 hover/选中）
  • 右面板: bbox 列表 + 编辑器（识别结果、ground truth）+ 预设字体列表
  • 拖拽框选新区域 → VLM 识别（MVP 走 mock）

4. ExportPage（导出）

  • XML: PASCAL VOC 格式
  • JSON: {dt_polys_rec: [{points, rec}, ...]}
  • HTML: 自包含，base64 图片 + SVG 叠加

状态管理（4 个 Zustand Store）

Store职责
useProjectStore图片 URL、尺寸、当前阶段、pipeline 状态
useDetectionStore检测多边形数组、置信度
useAnnotationStore字符 bbox、标签、ground truth
useUIStore选中、hover、面板开关

实施阶段

Phase内容产出
1Vite 脚手架 + 依赖 + 类型 + stores + 路由骨架可运行的空壳 app
2ImageDropzone + UploadPage图片上传功能
3ImageOverlay + MSW Mock + DetectPage检测可视化
4BboxList + BboxEditor + AnnotatePage标注编辑器
5XML/JSON/HTML 生成 + ExportPage导出功能
6国际化 + 错误处理 + GitHub Pages 部署完善

GPU 配置

A100 48GB，CUDA 12。当前阶段用 API 调用，后期可迁移到本地推理。
